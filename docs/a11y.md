# Accessibility policies (QCMS portal and admin)

**Status:** established by task 030 (portal accessibility pass) · **Standard:** WCAG 2.2 AA · **Inherited by:** admin app (031-035) and any future respondent-facing surface.

This document is the normative record of the flow-level accessibility behaviors a
component library cannot supply on its own: focus management across branch
changes, live-region announcements, keyboard/tab-order policy, and error UX. The
`@qcms/ui` renderer (028) supplies per-control semantics (labels, roles,
`aria-describedby` error wiring); the behaviors here are owned by the host app
(the portal today, the admin preview tomorrow) because only the host knows the
flow. ARCHITECTURE.md §11 is the commitment; this is the how.

## The focus-target primitive (@qcms/ui)

Every control the renderer draws is wrapped by the qcms-owned registry adapter in
a `display:contents` element carrying `id={questionId}` and
`data-qcms-field={questionId}` (task 030). The wrapper adds no box and no
accessibility-tree node, so the accessibility outline is unchanged; it exists
only so a host app can find "the DOM for question X" without reverse-engineering
each control type's internal markup. Everything below is built on this handle.

The honeypot (026) is deliberately NOT given this handle and stays `aria-hidden`
and out of the tab order: it must remain invisible to assistive tech.

## Focus management on branch changes

A forward-pass answer can insert or remove downstream questions (ADR-16). Policy:

- **Insertion (a branch reveals questions):** focus STAYS on the control the
  respondent just answered. We never yank focus forward to the new question - that
  would strand a keyboard or screen-reader user mid-thought. The insertion is made
  perceivable by an announcement (below), not by a focus move.
- **Removal of a question that does NOT hold focus:** nothing to do for focus; the
  removal is announced.
- **Removal of the question that DOES hold focus:** focus moves to a stable,
  sensible target so it is never dropped to `<body>`. Target order:
  1. the next question that was visible after the removed one and still is, in the
     pre-change document order (`nextFocusTargetAfterRemoval`);
  2. if none follows, the step heading (`h1`, made programmatically focusable).
- **Focus preservation across re-render:** re-rendering the step from a new flow
  projection can remount the focused control. If the focused question survives the
  projection but the remount dropped focus, the host restores focus to that same
  question. Net effect: focus "stays put" even though the DOM was rebuilt.

Implementation: `apps/portal/lib/a11y.ts` (`diffFlow`, `nextFocusTargetAfterRemoval`,
`focusQuestion`, `questionIdOf`) driven by an effect in
`apps/portal/components/step-flow.tsx`. The question that holds focus is captured
at the instant an answer post returns (before the re-render), so the post-render
effect knows what to recover.

## Live-region announcements

A single visually-hidden `aria-live="polite"` region (the "announcer",
`data-testid="flow-announcer"`) carries flow narration; error summaries use
`role="alert"` (assertive). All strings come from the shell i18n catalog
(`apps/portal/lib/i18n/en.ts`, single locale) - never hard-coded in components.

| Event | Politeness | Message |
| --- | --- | --- |
| Step change | polite | `Step {current} of {total}: {title}` (title read from the step `h1`) |
| Branch insertion | polite | `1 question was added below.` / `{n} questions were added below.` |
| Branch removal | polite | `1 question was removed.` / `{n} questions were removed.` |
| Blocked submit | assertive (`role="alert"`) | error summary heading + one entry per missing question |
| Answer rejected (422) | via the control's own error slot | server message in `aria-describedby` (028) |
| Submission success | page navigation | the `/done` heading announces on load; no live region needed |

Priority when several changes coincide in one projection: a step change outranks a
branch change (it is the larger context shift); otherwise an insertion outranks a
removal. One announcement per projection - we never stack messages.

## Keyboard traversal and tab order

- The full flow is completable keyboard-only: no pointer-only affordance exists.
- A skip link (`Skip to content` -> `#portal-main`) is the first focusable element
  and becomes visible on focus.
- **Tab order is DOM order.** Because insertion keeps focus on the answered
  control and the inserted question is placed after it in the document, the new
  question is simply the next Tab stop - no tabindex management, no focus trap.
  Positive `tabindex` is never used anywhere.
- No keyboard traps: every composite control (react-aria radio group, number
  field, date picker, select) is enter-and-leave by Tab per its ARIA pattern.

## Focus visibility (WCAG 2.2 - 2.4.11)

The vendored `@qcms/ui` controls carry their own focus rings. Every other
focusable element (skip link, error-summary "jump to field" links, a step heading
focused after a branch removal) gets a guaranteed high-contrast outline from a
zero-specificity `:where(...):focus-visible` rule in `globals.css`, using
`--color-focus-ring`, which clears 3:1 against its background in BOTH light and
dark themes (ADR-26).

## Error UX (WCAG 3.3)

- A blocked submit renders an error SUMMARY above the first field
  (`role="alert"`, focusable), and focus moves to it so it is announced.
- Each summary entry is a link that moves focus to its offending field via the
  focus-target handle (`focusQuestion`), scrolling it into view. This is the
  standard "error summary that jumps to the field" pattern.
- Per-field errors are the server's (the authority); the renderer surfaces each in
  the control's error slot, associated by `aria-describedby` (028). The portal
  never invents client-side validation copy beyond the catalog.

## Reduced motion (WCAG 2.3.3)

`globals.css` honors `prefers-reduced-motion: reduce` with the conservative reset
(transitions/animations collapse to near-instant, `scroll-behavior: auto`). The
design uses little motion; this guarantees none is imposed on users who opted out.

## Automated enforcement (CI)

- **axe** (`@axe-core/playwright`) runs against EVERY portal state in the fixture
  walkthroughs, including post-branch-insertion, post-branch-removal, and the
  blocked-submit error-summary state - not just first render. Zero violations is
  the gate.
- **Lighthouse** (`lighthouse`, accessibility category, threshold 100) runs on the
  page-load states that a fresh navigation can reach: entry, the flow page (session
  cookie injected), an error message screen, and completion. Interactive-only
  states (branch changes, the blocked-submit summary) are covered by axe, which
  runs inside the real, cookie-bearing browser context. The cookie-context
  rationale (why Lighthouse needs the session cookie injected as a request header)
  is documented in the header of `apps/portal/e2e/a11y-lighthouse.pw.ts`.
- **Reduced-motion and 200% zoom** smoke checks run on the flow page.

## Manual pass (human-in-the-loop)

Automated tools catch roughly half of WCAG issues. Each release also runs a manual
screen-reader pass (NVDA + VoiceOver) against the kitchen-sink form, driven by
`docs/a11y-manual-pass-checklist.md` and logged as `docs/a11y-pass-<date>.md`.
Zero open severity-1 issues is a Stage 7 gate.

## Inheritance note for the admin app (031-035)

The admin preview renders through the same `@qcms/ui` renderer, so the
focus-target primitive is already present there. When the admin builds its own
flow preview it MUST reuse these policies (announcer region, focus-on-branch,
error summary, skip link, focus-visible fallback) rather than reinventing them;
admin has its own axe gate but the flow-level behaviors are defined here.
